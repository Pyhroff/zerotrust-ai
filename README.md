# ZeroTrust AI

> **Cryptographic accountability for AI systems — zero-knowledge proofs for prompt evidence, safety auditing, and machine unlearning.**

ZeroTrust AI implements three ZKP primitives that close a governance gap no existing zkML toolkit addresses: not *how* a model computed an output, but *whether* the system behaved honestly around that computation — anonymously, verifiably, and without disclosing proprietary data.

```
python main.py prompt demo --reveal     # anonymous AI evidence (whistleblower mode)
python main.py audit demo               # hidden-suite safety certification
python main.py unlearn demo             # verifiable machine unlearning (GDPR)
python main.py tamper-demo              # soundness: 8 tampering attacks, all rejected
```

---

## Motivation

Current AI accountability is assertion-based:

| Claim | State of the art | What ZKP enables |
|-------|-----------------|-----------------|
| "Our model passed safety evaluations" | Self-reported PDF | Cryptographic certificate — verifiable without seeing the test suite |
| "We deleted your data from our model" | Privacy policy promise | Proof of unlearning procedure on committed weights |
| "This AI gave me harmful advice" | Screenshot (forgeable) | Operator-signed session with anonymous ownership proof |

This work is motivated by the EU AI Act (Art. 9, 72), GDPR Art. 17 machine unlearning obligations, and the emerging zkML literature — none of which yet provide a practical toolkit for the *accountability layer* above verifiable inference.

## Protocol Specifications

### Cryptographic Setup

Let $p$ be the RFC 3526 2048-bit safe prime ( $p = 2q + 1$, $q$ prime). The implementation works in the **prime-order subgroup** of $\mathbb{Z}_p^\*$ of order $q$, generated by $h = g^2 \bmod p$ (squaring maps the full-group generator $g=2$ into the $q$-order subgroup, guaranteeing all public keys satisfy $y^q \equiv 1 \pmod{p}$).

All proofs use the **Fiat-Shamir heuristic** to convert interactive Schnorr $\Sigma$-protocols into NIZKPs in the random oracle model. Challenge hashes are prefixed with a domain separation label `ZeroTrust-Schnorr-v1` to prevent cross-protocol attacks.

### Module 1 — `zk-prompt`: Anonymous AI Evidence

**Problem.** A user interacts with an AI system and receives a response they later want to prove is authentic. Revealing their identity in doing so may expose them to retaliation (whistleblower) or privacy violation. A forged screenshot proves nothing. We want: *prove the operator produced this (prompt, response) pair for this pseudonym, without revealing the pseudonym's real identity or the prompt content.*

**Parties.** Operator $\mathcal{O}$, User $\mathcal{U}$, Verifier $\mathcal{V}$ (any third party).

**Protocol.**

*Setup:*
- $\mathcal{O}$ generates keypair $(sk_\mathcal{O}, pk_\mathcal{O}) \in \mathbb{Z}_q \times \langle h \rangle$. Publishes $pk_\mathcal{O}$.
- $\mathcal{U}$ generates an **anonymous** keypair $(sk_\mathcal{U}, pk_\mathcal{U})$. Never linked to real identity.

*Session:*

$$C_P = \text{SHA-256}(P \| r), \quad r \leftarrow_R \{0,1\}^{256}$$

$\mathcal{U}$ sends $(C_P, pk_\mathcal{U})$ to $\mathcal{O}$. $\mathcal{O}$ responds with the model output $R$ and signs:

$$\sigma_\mathcal{O} = \text{Schnorr.Sign}\!\left(sk_\mathcal{O},\ \text{SHA-256}(C_P \| R \| pk_\mathcal{U} \| t \| \eta)\right)$$

where $t$ is a Unix timestamp and $\eta \leftarrow_R \{0,1\}^{128}$ is a per-session nonce preventing signature replay.

*Proof generation:* $\mathcal{U}$ produces a NIZKP of knowledge of $sk_\mathcal{U}$ s.t. $pk_\mathcal{U} = h^{sk_\mathcal{U}}$, bound to the session hash. Published proof: $(C_P, R, pk_\mathcal{U}, t, \eta, \sigma_\mathcal{O}, \pi_{id})$. The prompt $P$ and real identity remain hidden.

*Whistleblower mode:* $\mathcal{U}$ optionally opens $C_P$ by revealing $(P, r)$. Any party checks $\text{SHA-256}(P \| r) = C_P$.

**Security properties.**

| Property | Argument |
|----------|----------|
| Zero-knowledge | Simulator: given $pk_\mathcal{U}$, sample $(R', s') \leftarrow_R \mathbb{Z}_q^2$, set $e' = \text{challenge}(\ldots)$, output $(R', s')$. Indistinguishable from real proof in ROM. |
| Soundness | Breaking the identity proof extracts $sk_\mathcal{U}$ by rewinding (DLOG hardness). |
| Prompt hiding | $C_P$ is computationally hiding under SHA-256 preimage resistance. |
| Replay resistance | Per-session nonce $\eta$ makes the signed message unique even for repeated identical queries. |
| Operator binding | $\sigma_\mathcal{O}$ is a Schnorr signature over the session; forging it requires breaking DLOG. |

### Module 2 — `zk-audit`: Hidden-Suite Safety Certification

**Problem.** An auditor holds a red-team test suite $\mathcal{T} = \{(x_i, c_i)\}_{i=1}^n$ (inputs and behavioural constraints). Revealing $\mathcal{T}$ to the model operator would let them overfit to it; revealing it to the public would allow adversarial exploitation. We want: *prove a model passed every test in $\mathcal{T}$ without disclosing any $(x_i, c_i)$.*

**Protocol.**

*Commitment:*

$$\ell_i = \text{SHA-256}(x_i \| \texttt{0x00} \| c_i), \quad \text{root} = \text{MerkleRoot}(\ell_1, \ldots, \ell_n)$$

The auditor publishes `root` and $n$. The test cases remain secret.

*Operator evaluation:* For each test, $\mathcal{O}$ receives $(x_i, c_i, i, \text{root})$, evaluates the model, and signs:

$$\sigma_i = \text{Schnorr.Sign}\!\left(sk_\mathcal{O},\ \text{SHA-256}(\text{root} \| i \| \ell_i \| \text{result}_i)\right)$$

Note `root` is bound into every $\sigma_i$ — signatures from a different suite cannot be transplanted.

*Proof:* For each $i$, the proof bundles $(\sigma_i, \text{result}_i, \pi_i^{\text{Merkle}})$ where $\pi_i^{\text{Merkle}}$ is the Merkle inclusion path proving $\ell_i$ is in the committed tree. Verifier checks:

1. Each $\sigma_i$ is valid under $pk_\mathcal{O}$ with `result = PASS`
2. Each $\pi_i^{\text{Merkle}}$ opens against `root`
3. The leaf hash in $\sigma_i$ matches the leaf in $\pi_i^{\text{Merkle}}$

Signatures are batch-verified via random linear combination (see §Batch Verification), reducing $n$ modular exponentiations to a constant number.

**Security properties.**

| Property | Argument |
|----------|----------|
| Test suite hiding | Merkle commitment is collision-resistant; individual leaves are SHA-256 preimages. |
| Result binding | Suite root in $\sigma_i$ prevents transplanting a PASS from a different suite. |
| Completeness binding | Merkle tree forces $n$ leaves; auditor cannot exclude failing tests without changing `root`. |
| Operator accountability | Forging a PASS signature requires breaking DLOG on $pk_\mathcal{O}$. |

### Module 3 — `zk-unlearn`: Verifiable Machine Unlearning

**Problem.** Under GDPR Art. 17, a data subject $\mathcal{D}$ can request deletion of their data from a trained model. The operator must comply but has no incentive to prove it. We want: *prove that a specific public unlearning procedure was applied to committed model weights to remove a committed data point — without revealing the weights or the data.*

**Formal claim.** Let $\mathcal{P} = \texttt{GAU-v1}$ be the public procedure (gradient ascent + retain fine-tune). Given:

$$C_M = \text{Commit}(\theta),\quad C_d = \text{Commit}(d),\quad C_{M'} = \text{Commit}(\theta')$$

prove: $\theta' = \mathcal{P}(\theta, d)$ and the requester knows $d$.

**Protocol.**

*Commitments:*

$$C_X = \text{SHA-256}(X \| r_X),\quad r_X \leftarrow_R \{0,1\}^{256}$$

for $X \in \{\theta, d, \theta'\}$. Weights $\theta$ are quantized to int8 before hashing (deterministic byte representation).

*Operator attestation:*

$$\sigma_\mathcal{O} = \text{Schnorr.Sign}\!\left(sk_\mathcal{O},\ \text{SHA-256}(C_M \| C_d \| C_{M'} \| \texttt{GAU-v1})\right)$$

Procedure ID is bound into the signed message — the attestation is invalid for any other procedure.

*Data knowledge proof:* $\mathcal{D}$ generates NIZKP of knowledge of $sk_\mathcal{D}$ s.t. $pk_\mathcal{D} = h^{sk_\mathcal{D}}$, using $C_d$ as the statement binding the proof to this specific request.

*Weight delta:* Operator computes $\Delta = \|\theta - \theta'\|_\infty$ and includes it in the attestation. Verifiers can reject implausibly large or zero deltas (a zero delta would indicate the procedure was not run).

**Scope of the claim.** This proof establishes that the procedure was executed on the committed weights. It does **not** prove that the data point has zero influence in $\theta'$ — this is an open problem in the machine unlearning literature. The ZKP makes the approximation *transparent*: verifiers know exactly what guarantee they are getting.

## Cryptographic Design Decisions

### Why Schnorr over ECDSA?

Schnorr signatures support **batch verification** and **linear signature aggregation**, neither of which holds for ECDSA. For `zk-audit` with large test suites, batch verification saves $\approx n-1$ modular exponentiations. Schnorr proofs also have tighter security reductions in the ROM.

### Why DLP over a safe prime, not an elliptic curve?

Elliptic curve groups would be more efficient (256-bit scalars vs 2048-bit integers). The safe-prime choice keeps the implementation stdlib-only with no curve arithmetic, making the cryptography auditable without specialist knowledge. A production deployment should use Ristretto255 or BLS12-381.

### Why hash commitments over Pedersen?

Pedersen commitments are homomorphic, enabling arithmetic proofs over committed values (useful for future weight-arithmetic proofs). SHA-256 commitments are simpler, computationally binding under SHA-256 collision resistance, and sufficient for the current claim (procedure was run on *these* committed weights, not a proof about what the weights *are*). The architecture is designed to swap in Pedersen when the prover moves to a zkSNARK circuit.

### Batch Verification

For $n$ signatures $\{(pk_i, m_i, (R_i, s_i))\}$, draw random scalars $r_i \leftarrow_R \mathbb{Z}_q$ after receiving all signatures, then check:

$$\prod_{i=1}^{n} \left(h^{s_i} \cdot pk_i^{e_i} \cdot R_i^{-1}\right)^{r_i} \equiv 1 \pmod{p}$$

Soundness: an adversary who could produce a batch-passing forgery without a valid individual signature would need to predict the $r_i$ before they are drawn, which happens with probability $q^{-1}$ per attempt.

### Fiat-Shamir Domain Separation

Every challenge hash is prefixed with `ZeroTrust-Schnorr-v1 || len(label)`. Without domain separation, a proof produced in `zk-prompt` could be presented as an identity proof in `zk-unlearn`, since both prove knowledge of a discrete log. The label makes the challenge hash output protocol-specific.

## Security Properties Summary

| Attack | Addressed by |
|--------|-------------|
| Forged operator signature | Schnorr unforgeability under DLOG hardness |
| Prompt content leak | Hiding commitment $C_P = \text{SHA-256}(P \| r)$ |
| Identity deanonymisation | Schnorr ZK — simulator exists; proof reveals nothing beyond $pk_\mathcal{U}$ |
| Session replay | Per-session 128-bit nonce $\eta$ bound into $\sigma_\mathcal{O}$ |
| Cross-protocol proof transplant | Fiat-Shamir domain separation label |
| Small-subgroup attack on public key | Subgroup membership check: $y^q \equiv 1 \pmod{p}$ |
| Timing oracle on commitment open | `hmac.compare_digest` (constant-time byte comparison) |
| Audit result transplant | Suite root bound into every $\sigma_i$ |
| Merkle second-preimage | SHA-256 collision resistance; leaf/node domain distinction |
| Unlearning procedure substitution | Procedure ID string bound into operator attestation |


**Dependency surface:** `click` (CLI), `rich` (terminal output), `numpy` (neural net). All cryptographic code uses Python stdlib only — `hashlib`, `hmac`, `secrets`, `struct`.

## Trust Model

ZeroTrust AI uses an **operator-participation model**. The operator must hold a signing keypair and sign their outputs. This is the same trust assumption as TLS: you trust the certificate authority and cryptography handles everything above it.

```
                  publishes pk_op
Operator ─────────────────────────────────► Anyone

Operator ──signs──► (prompt_commit, response, pk_user, t, η)
                                │
                      User holds session + signature
                                │
                      User runs NIZKP of identity
                                │
                         Proof published ──► Verifier checks:
                                              1. Schnorr(pk_op, session) valid
                                              2. NIZKP(pk_user) valid
                                              3. No prompt, no real identity revealed
```

**What the operator cannot learn:** who the user is beyond their pseudonym `pk_user`.

**What breaks if the operator cheats:** the operator could sign false session records. This is equivalent to a CA misissuance — detectable in a public transparency log. A future extension could require operators to publish signed session commitments to a public ledger, enabling auditors to detect inconsistencies.

## Usage

```bash
# zk-prompt: anonymous evidence
python main.py prompt demo
python main.py prompt demo --reveal            # open the prompt (whistleblower mode)
python main.py prompt demo --output proof.json # export proof for offline verification

# zk-audit: hidden safety certification
python main.py audit demo
python main.py audit demo --output audit.json

# zk-unlearn: GDPR data removal proof
python main.py unlearn demo
python main.py unlearn demo --output unlearn.json

# soundness check: tamper with every proof field
python main.py tamper-demo
```

Exported proofs are self-contained JSON — pass them to `verify_proof()` directly:

```python
import json
from zerotrust.prompt.prover import verify_proof

with open("proof.json") as f:
    data = json.load(f)

result = verify_proof(data["proof"])
print(result["valid"])   # True
```

## Limitations

**Proof of procedure, not proof of influence.** `zk-unlearn` proves the GAU procedure was applied to the committed model. It does not prove the data point has zero influence in the resulting model — that is an open problem. See [Thudi et al., 2022](https://arxiv.org/abs/2205.02284) for a formal treatment of why exact unlearning verification is hard.

**Group choice.** The 2048-bit MODP group provides ~112-bit security, comparable to RSA-2048. A production system should use a 256-bit elliptic curve group (Ristretto255) for equivalent security with ~8x faster operations.

**Quantized weights.** The neural net uses int8 quantization before commitment. Quantization error means $\text{Commit}(\theta_\text{float})$ and $\text{Commit}(\theta_\text{int8})$ are different objects. In production the commitment should be over the canonical floating-point bit representation (e.g., IEEE 754 big-endian) to avoid ambiguity.

**Operator trust.** As with TLS, the system does not prevent a malicious operator from signing false records. Transparency logs or multi-operator threshold signing could reduce this assumption — these are natural extensions.
